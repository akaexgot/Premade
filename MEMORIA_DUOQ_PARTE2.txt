13. DESARROLLO POR FASES
================================================================

El proyecto se dividió en 6 fases principales (pasos), cada una
construida sobre la anterior:

FASE 1 — ESTRUCTURA BASE
- Creación de la estructura de carpetas Clean Architecture
- Configuración del tema (AppTheme, AppColors, AppTextStyles)
- Implementación de validadores de formularios
- Definición de clases de error (Failure)
- Configuración del archivo pubspec.yaml con todas las dependencias

FASE 2 — BASE DE DATOS Y SUPABASE
- Diseño completo del esquema SQL (schema.sql, 703 líneas)
- Creación de 16 tablas con relaciones, índices y constraints
- Implementación de Row Level Security (RLS) en 12 tablas
- Funciones SQL: calculate_match_compatibility, get_match_candidates
- Triggers automáticos de updated_at
- Datos iniciales: juegos, roles y rangos de LoL y Valorant
- Implementación de SupabaseService (singleton, 816 líneas)
- Configuración de variables de entorno (.env)

FASE 3 — AUTENTICACIÓN
- LoginPage y RegisterPage con validación de formularios
- SplashPage con detección de sesión activa
- Google OAuth (signInWithOAuth)
- AuthRepository, AuthRemoteDataSource
- Entidades Freezed: AuthEntity
- Use Cases: LoginUseCase, RegisterUseCase, LogoutUseCase
- Riverpod providers de auth

FASE 4 — PERFILES
- ProfileSetupPage (nickname, edad, país, bio, Discord)
- GameSelectionPage con selección de juego, rango y rol
- Subida de avatar con image_picker → Supabase Storage
- EditProfilePage y MyProfilePage
- ProfileRepository, ProfileRemoteDataSource
- Entidades Freezed: UserProfileEntity, GameEntity
- Use Cases: GetProfileUseCase, UpdateProfileUseCase, etc.

FASE 5 — MATCHING
- HomePage con stack de swipe cards (MatchCardWidget)
- Algoritmo de swipe: like/dislike/superlike con animaciones
- Detección de match mutuo en SupabaseService.swipeUser()
- getMatchCandidates con RPC SQL + fallback manual
- MatchingRepository, MatchingRemoteDataSource
- Entidades Freezed: MatchingEntity, MatchEntity
- Filtros por país y juego en swipe
- PublicProfilePage para ver perfil de candidatos

FASE 6 — CHAT Y SOCIAL
- ChatPage (lista de conversaciones)
- ChatDetailPage (mensajes + input)
- Supabase Realtime: subscribeToMessages() con StreamController
- Sistema social: amigos, solicitudes, bloqueos
- FriendsPage con tabs Amigos / Solicitudes
- ChatRepository, ChatRemoteDataSource
- Entidades Freezed: Conversation, Message, ConversationPreview
- 7 Use Cases de chat
- 15+ Riverpod providers de chat
- SearchPage con filtros

15. MANUAL DE INSTALACIÓN
================================================================

REQUISITOS PREVIOS:
- Flutter SDK >= 3.13.0 instalado y en el PATH
- Dart SDK >= 3.0.0 (incluido con Flutter)
- Android Studio o VS Code con extensión Flutter/Dart
- Python >= 3.0 (opcional, para scripts de generación)
- Git

PASO 1: Clonar el repositorio
  git clone <url-del-repositorio>
  cd premade

PASO 2: Configurar Supabase
  2.1. Ir a https://app.supabase.com → New Project
  2.2. Nombre: premade-production, Region: Europe (Frankfurt)
  2.3. En Settings → API, copiar:
       - Project URL
       - anon public key
  2.4. En SQL Editor → New Query, pegar y ejecutar:
       database/schema.sql (703 líneas)
  2.5. Verificar ejecución sin errores

PASO 3: Configurar variables de entorno
  Crear archivo .env en la raíz del proyecto:
  ------------------------------------------------
  SUPABASE_URL=https://xxxxx.supabase.co
  SUPABASE_ANON_KEY=eyJhbGciOiJI...
  ------------------------------------------------

PASO 4: Instalar dependencias Flutter
  flutter pub get

PASO 5: Generar código (Freezed + Riverpod)
  dart run build_runner build --delete-conflicting-outputs

PASO 6: Ejecutar la aplicación
  flutter run
  (Con emulador Android activo o dispositivo conectado)

VERIFICACIÓN:
  Ejecutar en SQL Editor de Supabase:
  SELECT tablename FROM pg_tables WHERE schemaname = 'public';
  → Debe devolver las 16 tablas
  SELECT * FROM public.games;
  → Debe mostrar League of Legends y Valorant

SOLUCIÓN DE PROBLEMAS:
- "relation already exists": DROP SCHEMA IF EXISTS public CASCADE; 
  CREATE SCHEMA public; (y volver a ejecutar schema.sql)
- "permission denied": Verificar RLS y políticas en Supabase Dashboard
- Error de build_runner: Eliminar carpeta .dart_tool y repetir

17. SEGURIDAD
================================================================

AUTENTICACIÓN:
- Delegada completamente a Supabase Auth (sistema probado en producción)
- Tokens JWT con expiración automática
- autoRefreshToken configurado para gestión segura de sesiones
- Contraseñas nunca almacenadas en la app ni en texto plano
- Variables sensibles en .env (no versionado en Git)
- service_role_key NUNCA expuesto en el cliente Flutter

ROW LEVEL SECURITY (RLS):
Habilitado en las 12 tablas principales. Garantiza que:
- Un usuario solo puede ver su propio perfil O perfiles públicos
  no bloqueados (no puede acceder a datos ajenos con SQL directo)
- Solo puede editar su propio perfil (UPDATE policy)
- Solo puede ver mensajes de conversaciones donde participa
- Solo puede ver sus propias notificaciones
- No puede ver usuarios que le han bloqueado (ni viceversa)

CONSTRAINTS DE BASE DE DATOS:
- age >= 13 (menores de edad no pueden registrarse)
- bio entre 10 y 500 caracteres (evita spam)
- CHECK user_id != target_user_id (no auto-swipe)
- CHECK user_id_1 < user_id_2 (evita duplicados en matches)
- Nicknames y emails únicos (UNIQUE constraints)
- FK con ON DELETE CASCADE para integridad referencial

SISTEMA DE BLOQUEOS:
- Usuarios bloqueados se excluyen en matching, búsqueda y chat
- Las políticas RLS verifican la tabla blocks en tiempo real
- El bloqueador y el bloqueado no se ven mutuamente

SISTEMA DE REPORTES:
- Tabla reports con estados: pending → under_review → resolved
- Usuarios reportados marcados con is_reported=TRUE
- Flujo de moderación previsto

SOFT DELETE:
- Usuarios eliminados tienen deleted_at != NULL
- Función cleanup_deleted_users() para hard delete diferido (30 días)
- Los candidatos de matching excluyen usuarios con deleted_at

GIT / CÓDIGO:
- .env en .gitignore (credenciales no versionadas)
- .env.example disponible con variables placeholder

19. INCIDENCIAS Y MEJORAS
================================================================

INCIDENCIAS CONOCIDAS Y RESUELTAS:

1. Doble ID de usuario (auth_id vs profile_id):
   Supabase Auth genera un UUID para auth (auth_id) diferente al
   UUID de la tabla public.users (profile_id). Se implementó
   getProfileId() con caché en SupabaseService para resolver el
   mapeo de forma eficiente.

2. Fallback en get_match_candidates:
   La función RPC SQL get_match_candidates a veces falla por
   permisos. Se implementó un fallback manual en Dart que
   reproduce la misma lógica con queries encadenadas.

3. Estado de amistad en perfil público:
   Se añadió getFriendshipStatus() que normaliza el orden de
   user_id_1/user_id_2 (siempre el menor primero) para evitar
   duplicados y garantizar unicidad de la relación.

4. Deduplicación de matches:
   CHECK constraint user_id_1 < user_id_2 garantiza que no
   pueden existir dos filas para el mismo par de usuarios.
   createMatch() ordena los IDs antes de insertar.
